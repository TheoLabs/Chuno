S1-1core-api 인프라 세팅BACKENDM
NestJS에 TypeORM+MySQL 연결과 설정 로딩을 붙여 이후 도메인이 얹힐 토대를 만든다.
  • 완료@nestjs/config ConfigModule(@configs) — 환경파일(.env.<NODE_ENV|local>) + configuration.ts + 커스텀 validate(누락 env 감지), ConfigsService(mysql/google/jwt getter)
  • 완료TypeOrmModule.forRootAsync — ConfigsService.mysql 주입, MySQL(mysql2) 연결(charset utf8mb4) + 라이프사이클 로깅(TypeORM 1.0.0)
  • 완료헬스체크 — setGlobalPrefix('api') 복원, GET /api/health → 200 실측 확인. DB 연결(ping)은 스코프 제외로 확정, 정적 {status:'ok'}로 충분
  • 완료graceful shutdown — main.ts에서 enableShutdownHooks() 호출 + onModuleDestroy에서 await datasource.destroy()
  • 결정마이그레이션 스크립트(migration:run) 미도입 결정 — 로컬/현재는 synchronize:true로 스키마 관리, CLI DataSource + synchronize:false 전환 작업 자체를 도입하지 않기로 함
완료 — 앱 부팅 시 DB 연결 성공, GET /api/health 200, ConfigModule·TypeORM·charset·graceful shutdown 등 핵심 인프라 완료. 마이그레이션 스크립트는 미도입 결정 — synchronize로 스키마 관리.

S1-3User / AuthIdentity 엔티티 + 마이그레이션BACKENDS
사용자와 소셜 신원을 저장할 스키마. 인증·프로필의 기반 애그리거트.
  • 완료User(nickname? unique·level?(RunnerLevel)·tier(RunnerTier, 기본 BRONZE)·profileImageFileId?·bio?·joinOn(CalendarDate)) + User.of() 팩토리(authIdentity·consents 조립)
  • 완료AuthIdentity — User와 1:1(@OneToOne + @JoinColumn userId) + provider enum(kakao/google/apple)
  • 재검증UserConsent(userId·type·legalDocumentId·documentVersion·agreedOn·revokeOn?) — legalDocumentId(LegalDocument 논리 FK, 물리 FK 아님) 추가, documentVersion(동의 시점 서버 스냅샷)은 유지 + @ManyToOne User(1:N) + revoke() LegalDocument 참조 반영 재작업 필요
  • 완료UserRepository(포트, DddRepository 상속) + find/count — 앱 부팅 시 DI 정상, 스키마 자동 생성, GET /api/health 200 실측
  • 완료sub 컬럼 반영 — provider의 외부 소셜 고유ID(Google/Apple sub, Kakao id), 토큰 검증 후 조회 키로 사용. 기존 (userId, provider) 복합 유니크는 UNIQUE(provider, sub)로 대체(같은 소셜 계정이 두 User에 붙는 것 방지) — User와는 계속 1:1(userId) S1-4에서 반영·실측
  • 결정마이그레이션 스크립트(generate/run) 도입 작업 자체를 하지 않기로 결정 — 로컬/현재는 synchronize:true로 스키마 관리
  • 완료UserConsentType에 MARKETING(선택 동의) 추가 S1-6 재검증에서 확인
  • 완료첫 소셜 로그인 = 온보딩 전 User 생성 — (provider, sub) 없으면 User+AuthIdentity 생성 시점엔 아직 닉네임/레벨 없음 → nickname·level nullable, onboardedOn(CalendarDate?, 온보딩 완료 시각) 추가. 온보딩(S1-10)에서 채움, AuthGate(S1-8)는 onboardedOn null 여부로 분기 S1-4에서 반영·실측
재검증 필요 — User·AuthIdentity(1:1)·UserRepository(포트), synchronize로 스키마 생성, 앱 부팅·스키마 생성은 기존에 실측 확인됐다. 마이그레이션 스크립트는 미도입 결정. sub/nullable nickname·level/onboardedOn 컬럼 적용은 S1-4에서 완료. 남음 — UserConsent에 legalDocumentId(LegalDocument 논리 FK) 컬럼 추가 반영 후 재검증(마이그레이션 스크립트는 미도입 결정 유지).

S1-4소셜 토큰 검증 + AUTH_DEV_MODEBACKENDM
provider별로 클라이언트가 넘긴 토큰을 검증해 신뢰 가능한 소셜 신원을 얻는다.
  • 완료provider별 verifier — google(google-auth-library verifyIdToken, audience 검증) / apple(jose JWKS + iss/aud 검증) / kakao(사용자 API) — 검증 결과에서 provider 외부 고유ID(sub)를 얻는다
  • 완료sub로 (provider, sub) AuthIdentity 조회 · 없으면 온보딩 전 User + AuthIdentity 생성(nickname/level null, onboardedOn null) — 있으면 연결된 User로 매핑
  • 완료AUTH_DEV_MODE: "dev:<sub>:<email>" 토큰으로 키 없이 로컬 테스트 — 플래그 on일 때만 동작(기본 off로 프로덕션 우회 차단)
  • 완료UNIQUE(provider, sub) 반영, 날짜 규칙 준수, DDD 레이어 분리(컨트롤러는 아직 없음 — S1-5에서 노출) code-reviewer 실측: build/lint OK, 테스트 12/12, 부팅+스키마 OK
완료 — dev 토큰과 실제 idToken 모두 검증되어 (provider, sub)로 올바른 User에 매핑되고, 첫 로그인은 온보딩 전 User가 생성된다. 실 소셜 키 E2E는 S1-12 키 발급 후로 명시. 후속 개선(완료를 막지 않음): 프로덕션 isProduction() fail-fast 다중 방어(AUTH_DEV_MODE) · resolveOrCreateUser 동시 첫 로그인 unique 충돌 시 upsert-on-conflict · Kakao id 누락 시 명시적 예외.

S1-6users/me APIBACKENDS

  • 재검증PUT /users/onboard(닉네임+레벨+consents:[{type,legalDocumentId}]) — 클라는 활성 문서 legalDocumentId만 전송, 서버가 해당 LegalDocument의 version을 UserConsent에 스냅샷 기록. 필수 동의 3종(TERMS·PRIVACY·LOCATION) 전부 포함 검증을 legalDocumentId 기준으로 재작업 필요 기존 consents:[{type,documentVersion}] 방식의 happy path·409/400 케이스는 legalDocumentId 도입 후 재검증. S1-11의 동의 서버 기록 요구사항 자체는 유효하나 페이로드 재검증 필요(선행: S1-15·S1-16)
  • 스코프외프로필 수정 PUT /users/me — 별도 태스크로 분리(추후)
  • 완료GET /users/check-nickname — @Query()로 전환(쿼리 ?nickname= 200 동작), @Length(2,12)(1자·13자·빈값 400 / 2·12자 통과), count 기반 고유성(존재→사용중, 미존재→가용) 실측. 직전 차단 버그(@Body()/GET, 길이검증 누락)는 해소됨 code-reviewer 재검증: build/lint OK, 부팅 후 실호출. 후속 노트(완료를 막지 않음): 응답 {usedCount} 의미 모호(→{available: boolean} 권장) · 닉네임 트림/정규화 없음(앞뒤 공백 통과, DTO 트림 권장) · soft-delete 유저 닉네임이 "가용"으로 보고될 수 있음(count는 deletedAt 제외, DB UNIQUE는 포함)
  • 완료JWT 가드(UserGuard)로 보호 — Bearer 액세스 JWT 검증(무토큰·잘못된 토큰 → 401 실측), 성공 시 User 객체(consents 포함)를 Context.USER에 심음; JwtModule 전역 등록(auth.module global:true)으로 UserModule 주입 정상 code-reviewer 실측: build/lint OK, 부팅+엔드투엔드
재검증 필요 — GET /users/me·check-nickname·JWT 가드는 이번 체인지셋과 무관해 완료 상태 유지. PUT /users/onboard는 consents 페이로드가 documentVersion 직접 전송 → legalDocumentId 참조로 바뀌어 재검증 필요(선행: S1-15·S1-16). PUT /users/me는 별도 태스크로 분리. 후속 노트(완료를 막지 않음): 재온보딩 시 도메인 409보다 닉네임중복 400이 먼저 나오는 순서 — 서비스에서 도메인 검증을 앞에 두면 해소 · 미사용 ConflictException import 정리 · GeneralUserService 빈 클래스(PUT /users/me 구현 시 채울 자리, 정리 권장) · UserGuard가 UserRepository 포트 대신 datasource 직접 조회(횡단 관심사라 완전 위반은 아니나 컨벤션과 다름) · UserGuard가 매 인증요청 consents 조인(필요한 라우트에서만 로드 고려).
