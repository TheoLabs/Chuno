이슈 보드/Step 1·MVP Step 1

Step 1 · 기반 & 인증

백엔드 골격 + 소셜 로그인 → 온보딩 → 홈 진입까지 실계정으로 도는 첫 뼈대. (총 12 이슈)

📋 MVP Step 1 ← 이슈 보드
S1-1core-api 인프라 세팅BACKENDM
NestJS에 TypeORM+MySQL 연결과 설정 로딩을 붙여 이후 도메인이 얹힐 토대를 만든다.
  • 완료@nestjs/config ConfigModule(@configs) — 환경파일(.env.<NODE_ENV|local>) + configuration.ts + 커스텀 validate(누락 env 감지), ConfigsService(mysql/google/jwt getter)
  • 완료TypeOrmModule.forRootAsync — ConfigsService.mysql 주입, MySQL(mysql2) 연결(charset utf8mb4) + 라이프사이클 로깅(TypeORM 1.0.0)
  • 완료헬스체크 — setGlobalPrefix('api') 복원, GET /api/health → 200 실측 확인. DB 연결(ping)은 스코프 제외로 확정, 정적 {status:'ok'}로 충분
  • 완료graceful shutdown — main.ts에서 enableShutdownHooks() 호출 + onModuleDestroy에서 await datasource.destroy()
  • S1-3 이월마이그레이션 스크립트(migration:run) — 엔티티가 아직 없어 현재는 synchronize:true, 첫 엔티티 도입 시점(S1-3)에 CLI DataSource + synchronize:false(isLocal() 가드)로 전환
완료 — 앱 부팅 시 DB 연결 성공, GET /api/health 200, ConfigModule·TypeORM·charset·graceful shutdown 등 핵심 인프라 완료. 마이그레이션(synchronize:false 전환 포함)은 S1-3에서 실행.
선행: —관련MVP
S1-2DDD 공용 기본형BACKENDS
바운디드 컨텍스트가 일관된 레이어 규약으로 자라도록, libs/ddd에 공용 도메인 기본형을 세운다.
  • 완료AggregateRoot 기본형(DddAggregate) — 도메인 이벤트 수집, libs/ddd/ddd-aggregate.ts
  • 완료DomainEvent 기본형 + 리포지토리·서비스 포트 골격 — libs/ddd/*
  • 완료build 통과
완료 — 공용 DDD 기본형 골격(AggregateRoot·DomainEvent·리포지토리/서비스 포트)이 libs/ddd에 컴파일되고, 후속 컨텍스트가 @libs/ddd로 import해 쓸 수 있다.
선행: S1-1관련MVP도메인
S1-3User / AuthIdentity 엔티티 + 마이그레이션BACKENDS
사용자와 소셜 신원을 저장할 스키마. 인증·프로필의 기반 애그리거트.
  • User(nickname·level·tier·profileImageUrl?·onboardedAt?) — Nickname/RunnerLevel VO
  • AuthIdentity(userId·provider·providerUserId) — User와 1:1(userId UNIQUE), 통합 소셜로그인(다중 연동) 미지원 · UNIQUE(provider, providerUserId)
  • UserConsent(userId·type·documentVersion·agreedAt·revokedAt?) — 약관/개인정보/위치 동의 이력, User와 1:N
  • UserRepository(포트) + TypeORM 구현·매퍼, 마이그레이션
완료 — 마이그레이션으로 테이블 생성, 동일 소셜 신원 중복 삽입이 유니크 제약으로 거부되고, 동의 기록이 type·documentVersion·agreedAt으로 저장된다.
선행: S1-1, S1-2관련MVP도메인
S1-4소셜 토큰 검증 + AUTH_DEV_MODEBACKENDM
provider별로 클라이언트가 넘긴 토큰을 검증해 신뢰 가능한 소셜 신원을 얻는다.
  • google: google-auth-library / apple: jose로 JWK 검증 / kakao: 사용자 API
  • 신원 → User 조회·없으면 생성(AuthIdentity 연결)
  • AUTH_DEV_MODE: "dev:<id>:<email>" 토큰으로 키 없이 로컬 테스트
완료 — dev 토큰과 실제 idToken 모두 검증되어 올바른 User로 매핑된다.
선행: S1-3관련MVP도메인
S1-5JWT + 리프레시 회전 + auth 엔드포인트BACKENDM
세션 토큰 발급/갱신/폐기. 리프레시 재사용 탐지로 탈취 방어.
  • 액세스 JWT 15m + opaque 리프레시(회전, sha256 해시 저장)
  • 리프레시 재사용 시 401 + 계열 무효화
  • POST /auth/social-login · /auth/refresh · /auth/logout
완료 — 로그인→토큰 발급, refresh로 회전, 이전 리프레시 재사용 시 401.
선행: S1-3, S1-4관련MVP도메인
S1-6users/me APIBACKENDS
온보딩·프로필에서 쓰는 내 정보 조회/수정과 닉네임 중복확인.
  • GET /users/me, PATCH /users/me(닉네임·레벨·onboardedAt)
  • GET /users/check-nickname (2~12자·고유 검증)
  • JWT 가드로 보호
완료 — 토큰으로 내 정보 조회·수정되고, 중복 닉네임이 거부된다.
선행: S1-3, S1-5관련MVP도메인
S1-7앱 네트워킹 / 상태관리 기반APPM
실 API 연동을 위한 공통 인프라 — 토큰 저장, 자동 갱신, 상태관리.
  • 완료상태관리 = Riverpod 확정 — main.dart ProviderScope, auth provider 배선(lib/features/auth/*)
  • 완료dio + flutter_secure_storage 도입, 실제 보안 저장(SecureKeyValueStore, Android encryptedSharedPreferences / iOS first_unlock)
  • 완료api_client 인터셉터 — 401 시 refresh 회전 후 원요청 재시도, single-flight(동시 401에 refresh 1회), 재시도 무한루프 방지, refresh 요청은 인터셉터 없는 전용 dio로 재귀 차단, 실패 시 토큰 삭제 + 세션만료 콜백
  • 완료TokenStore 보안 저장/조회/삭제, KeyValueStore 추상화로 단위테스트 대체
  • 완료Env(baseUrl + dart-define 오버라이드) · AppException(sealed, fromDio) · ApiResponse 공통 모델
  • 완료단위테스트 — 인터셉터(회전·재시도·single-flight·refresh 실패), TokenStore, AuthController flutter analyze 무이슈 + 33/33 테스트 통과(code-reviewer 실측)
완료 — 토큰이 보안 저장되고, 만료(401) 시 자동 refresh 회전 후 요청이 재개된다. 남음: 실 서버·baseUrl 미정으로 실 E2E(실제 401→refresh 왕복)는 백엔드 S1-5 확정 후 잔여 확인 · ApiResponse 파싱 일원화/AppException을 ApiClient 경계에 적용은 응답 스키마 확정(S1-5)·화면 연동(S1-8) 시 정리.
선행: —관련MVP
S1-8세션 / AuthGate 분기APPS
앱 시작 시 로그인·온보딩 상태에 따라 진입 화면을 결정한다.
  • 부팅 시 토큰 확인 → 로그인 / 온보딩미완 / 홈 분기
  • 로그아웃 시 스택 초기화(이미 UI 존재) + 토큰 삭제
완료 — 재실행 시 세션 유지로 홈 직행, 미온보딩 계정은 온보딩으로.
선행: S1-7, S1-5관련MVP화면
S1-9소셜 로그인 SDK 3종APPL
카카오·구글·애플 네이티브 로그인으로 idToken을 받아 백엔드에 전달.
  • google_sign_in · sign_in_with_apple · kakao SDK
  • iOS 네이티브엔 iOS 타입 클라이언트 필수(웹 클라 custom scheme 불가)

완료 — 3종 각각 실제 로그인 동의창까지 뜨고 백엔드 세션이 발급된다.
선행: S1-7, S1-12관련MVP화면
