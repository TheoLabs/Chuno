BO Step 1 · IAM 기반 + 법적문서 관리 재스코프

관리자 계정·역할·감사로그 도메인을 세우고, 사용자 트랙 S1-15/S1-16/S1-18의 "관리자 authz" 빈틈을 실제 role-RBAC로 메운다. (총 7 이슈, 전부 미착수)

📋 BO MVP Step 1 ← 백오피스 이슈
BO1-1Admin & IAM 바운디드 컨텍스트 도메인 설계DESIGNS
관리자 신원·역할·감사이력을 담을 새 바운디드 컨텍스트를 도메인 문서에 설계한다.
  • 신규 "Admin & IAM" 컨텍스트 카드 추가
  • AdminAccount·Role·AdminAuditLog 엔티티 카드 + #ent-adminaccount·#ent-role·#ent-adminauditlog 앵커
  • role-only RBAC 불변식 — Permission 엔티티 없음, Role(SUPER_ADMIN·OPERATOR·SUPPORT)이 곧 역할 단위 게이팅 경계
  • 감사로그 append-only + 논리참조(대상 엔티티에 물리 FK 아님) 규칙 명문화
  • ERD 노드·엣지 반영(엔티티 박스 관통 없이, 참조는 점선)
완료 — 도메인 문서에서 AdminAccount·Role·AdminAuditLog 3개 엔티티, role-only RBAC 불변식, 감사 append 규칙을 조회할 수 있고, #ent-adminaccount 딥링크가 정상 동작한다.
선행: —관련MVP도메인
바로가기📦 AdminAccount🗺️ MVP
BO1-2AdminAccount · Role 엔티티 + 리포지토리 + seedBACKENDM
외부 IdP 신원 기반 관리자 계정과 역할을 DDD 레이어로 구현하고 최초 super admin을 심는다.
  • modules/admin/ 신규 컨텍스트
  • AdminAccount(외부 IdP 신원: provider+subject, passwordHash 없음, status, lastLoginAt, roleIds[])
  • Role(SUPER_ADMIN·OPERATOR·SUPPORT)
  • UNIQUE(provider, subject)
  • seed super admin — 직접 주입(S1-15 seed 방식 재사용)
  • synchronize로 스키마 관리
완료 — 부팅 시 스키마가 생성되고, super admin·role seed가 DB에서 조회된다.
선행: BO1-1, S1-2관련MVP도메인

BO1-3관리자 SSO 인증 + AdminGuard + role 게이팅BACKENDM
관리자는 외부 IdP 토큰을 검증해 별도 admin JWT를 발급받고, role 기반 가드로 API를 보호한다.
  • POST /api/admins/auth/login(외부 IdP 토큰 검증 → admin JWT) · /refresh · /logout
  • 별도 JWT 시크릿 · 짧은 만료(S1-5 리프레시 회전 패턴 재사용)
  • AdminGuard + ContextKey.ADMIN
  • @RequireRoles(...) 데코레이터 — role 기반, permission-key 아님
  • SUSPENDED 관리자 거부
완료 — 관리자 자격으로만 /api/admins/* 접근 가능하고, 역할 미달 시 403.
선행: BO1-2관련MVP도메인

BO1-5관리자 법적문서 관리 API — S1-16 POST/PUT 재스코프BACKENDM
/api/admins/legal-documents — 초안 등록·확정을 role-RBAC로 보호하는 관리자 전용 엔드포인트로 재스코프한다.
  • GET /api/admins/legal-documents(전체 status·버전이력·DRAFT/ARCHIVED 포함)
  • POST(초안)
  • PUT /:id(확정·expectedActivateOn, PATCH 금지)
  • AdminGuard + OPERATOR 이상 role
  • 감사 기록
완료 — 관리자만 초안 등록·예약설정할 수 있고, 공개 GET은 active만 노출한다. S1-16의 "후속/최소 인증" 항목이 실 role-RBAC로 대체돼 재검증을 통과한다.
선행: BO1-3, S1-15, S1-16관련MVP도메인
