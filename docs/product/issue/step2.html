  • 결정TypeORM 매핑, synchronize로 스키마 관리(마이그레이션 미도입, Step1과 동일 결정)
통과 — Room·Participant 스키마·물리/논리 FK·직접컬럼·RoomStatus VO·도메인 메서드 create·join·leave·cancel(+불변식)·orphan 삭제·RoomRepository 포트·synchronize 실측 완료(build/lint 통과), create가 방장을 isHost로 포함하고 participants가 애그리거트 단위로 로드된다. (markStarting·markLive는 S2-3에서 구현 완료 — 이월 해소, participants 전용 로드 경로는 S2-2 소비 시점 적용.)

  • 완료Room 도메인 메서드 markStarting(RECRUITING→STARTING, 2명 미만→CANCELLED)·markLive(STARTING→LIVE), 멱등 구현(S2-1에서 이월) — 스케줄러(BullMQ, −10초/정각)가 호출
  • 완료STARTING 전환 시점(−10초)에 2명 미만이면 → CANCELLED(soft) 목록 비노출·통지
  • 스코프외LIVE 시 Race 생성은 Step3 경계(여기선 이벤트까지)
  • 완료방별 결정적 jobId로 잡 등록 — 방 취소·삭제 시 잡 취소·재등록 (시각변경 재등록은 시각변경 API 부재로 N/A, 스케줄러 자체는 재등록 지원)
  • 완료RoomSchedulerProcessor가 잡 소비 → 도메인 전환 → 도메인 이벤트 발행(RoomStarting·RoomLive·RoomCancelled)
  • 완료멱등성(중복·재시작 이중전환 방지)
완료 — RoomScheduler가 방별 결정적 jobId로 BullMQ 지연잡 2개(scheduledStartOn−10초에 markStarting / 정각에 markLive)를 등록하고, 방 취소·삭제 시 잡을 취소·재등록한다(시각변경 API는 아직 없어 N/A). RoomSchedulerProcessor가 잡을 소비해 도메인 전환 후 RoomStarting·RoomLive·RoomCancelled 이벤트를 발행하며, markStarting/markLive는 멱등(재시작·중복 잡에도 이중전환 없음)이 실측됐다. tsc 0에러·jest 32/32·실부팅 성공.
