Step 2 · 방 & 실시간 로비

방 생성·참가와 실시간(socket.io) 로비. 예약 시각 대기까지. (총 12 이슈)

📋 MVP Step 2 ← 이슈 보드
S2-1Room 애그리거트 + Participant + 스키마BACKENDL
한 번의 경주를 위한 모집 단위. 목표·시작시각·정원과 참가자를 담는 일관성 경계.
  • 완료Room(hostUserId·name·targetDistance·limitMinutes·maxParticipants·scheduledStartOn·RoomStatus·participants, DddAggregate)
  • 완료Participant(userId·joinedOn·isHost, 애그리거트 내부 자식) 필드·isHost·Room 물리FK·User 논리FK(userId plain) 실측
  • 완료targetDistance·limitMinutes는 Room 직접 컬럼, RoomStatus(RECRUITING·STARTING·LIVE·FINISHED·CANCELLED)만 VO(enum)
  • 완료도메인 메서드 create(방장 자동 첫 Participant)·join·leave·cancel + 불변식(정원·시작후 join불가·중복·방장leave=취소)
  • 이월markStarting(RECRUITING→STARTING, 2명미만→CANCELLED)·markLive(STARTING→LIVE)는 예약 스케줄러가 트리거하는 시점에 필요 → S2-3에서 구현(사용자 결정)
  • 완료RoomRepository(포트, DddRepository) — participants 애그리거트 단위 로드. find의 relations 파라미터로 로드 지원, 저장은 cascade로 애그리거트 단위 O(실측) participants를 항상 로드하는 전용 경로(findByIdWithParticipants) 보장은 소비 시점(S2-2 Room API)에 적용 — 사용자 결정으로 S2-1 스코프외
  • 결정TypeORM 매핑, synchronize로 스키마 관리(마이그레이션 미도입, Step1과 동일 결정)
통과 — Room·Participant 스키마·물리/논리 FK·직접컬럼·RoomStatus VO·도메인 메서드 create·join·leave·cancel(+불변식)·orphan 삭제·RoomRepository 포트·synchronize 실측 완료(build/lint 통과), create가 방장을 isHost로 포함하고 participants가 애그리거트 단위로 로드된다. (markStarting·markLive는 S2-3 예약 스케줄러 트리거 시점으로 이월, participants 전용 로드 경로는 S2-2 소비 시점 적용 — 둘 다 스코프 조정.)
선행: S1-2, S1-3관련MVP도메인

S2-2Room API (생성·탐색·참가·나가기·삭제)BACKENDM
방 생성·탐색·참가·나가기·삭제. 불변식을 서비스에서 강제한다.
  • POST /rooms(name·targetKm·limitMinutes·maxParticipants·scheduledStartOn, 미래시각만)
  • GET /rooms(공개목록+필터 거리·제한시간·빈자리, 상태필터, 임박순=scheduledStartOn asc, 좌표 미전송)
  • POST /rooms/:id/join · /leave
  • DELETE /rooms/:id(방장·RECRUITING만)
  • 불변식 — 정원초과·시작후 join 금지(늦참)·중복 join 금지
  • 결정방장 leave = 방 취소(CANCELLED)
  • 위반 시 409/400, PATCH 미사용
  • 스코프외관리자 강제취소/삭제·운영은 백오피스 BO4-1(/api/admins/rooms, PUT :id/cancel·DELETE). 여기 DELETE /rooms/:id는 방장 본인·RECRUITING 한정
완료 — 목록·생성·참가·나가기·삭제가 동작하고, 정원초과·늦참·중복참가·비방장삭제가 명확히 거부된다.
선행: S2-1관련MVP화면
바로가기📦 Room🎨 홈🎨 방 생성🎫 백오피스 BO4-1🗺️ MVP
S2-3예약 스케줄러 (동시출발 트리거)BACKENDM
예약 시각에 방 상태를 자동 전환. 인원 미달이면 취소. 동시출발의 트리거.
  • 결정BullMQ 지연잡 — scheduledStartOn(서버 KST)은 출발(LIVE) 시각. scheduledStartOn − 10초 도달 시 RECRUITING→STARTING(10초 카운트다운 시작·join 마감·이 시점 2명 미만이면 CANCELLED, 카운트다운 미개시), scheduledStartOn 도달 시 STARTING→LIVE(출발). Δ=10초 고정
  • 이월Room 도메인 메서드 markStarting(RECRUITING→STARTING, 2명 미만→CANCELLED)·markLive(STARTING→LIVE), 멱등 — 이 태스크에서 구현(S2-1에서 이월), 스케줄러(BullMQ, −10초/정각)가 호출
  • STARTING 전환 시점(−10초)에 2명 미만이면 → CANCELLED(soft) 목록 비노출·통지
  • 스코프외LIVE 시 Race 생성은 Step3 경계(여기선 이벤트까지)
  • 방 삭제·취소·시각변경 시 잡 취소·재등록
  • 도메인 이벤트 발행(RoomStarting·RoomCancelled)
  • 멱등성(중복·재시작 이중전환 방지)
완료 — scheduledStartOn − 10초에 자동 RECRUITING→STARTING(join 마감·카운트다운 시작, 이 시점 2명 미만이면 CANCELLED), scheduledStartOn(Δ=10초 후)에 STARTING→LIVE, 이벤트 발행, 재시작에도 이중전환이 없다.
선행: S2-1, S2-2, S2-5관련MVP도메인
바로가기📦 Room📦 상태 머신🗺️ MVP
S2-4로비 socket.io 게이트웨이 (+서버시각 소스)BACKENDM
로비 참가자에게 입퇴장·상태전환을 실시간 브로드캐스트. 클라 카운트다운을 위한 서버시각도 함께 제공.
  • JWT 핸드셰이크 인증(Step1 시크릿)
  • 스코프외제재(SUSPENDED/BANNED) 게이팅은 백오피스 BO2-4가 이 핸드셰이크에 크로스컷으로 추가(제재 유저 소켓 거부) — 이 태스크는 인증 훅 지점만 확보
  • 방 룸 조인·이탈, 참가자격 검증(해당 방 Participant만)
  • 브로드캐스트 participantJoined·participantLeft·roomStatusChanged·roomCancelled
  • 핸드셰이크/응답에 서버 now(serverTime, KST·epoch) 제공 — S2-9 오프셋 보정용
  • S2-5 이벤트 구독 → 브로드캐스트 배선
완료 — 참가/이탈·상태전환이 같은 방 클라이언트에 즉시 반영되고, 미인증 소켓은 거부되며, 클라가 서버시각 오프셋을 획득할 수 있다. 제재 게이팅 삽입 지점(핸드셰이크 인증 훅)이 열려 있다.
선행: S2-1, S2-5관련MVP화면

S2-5Room 도메인 이벤트 (인프로세스)BACKENDS
컨텍스트 간 연결을 위한 이벤트 발행. MVP는 인프로세스로.
  • RoomCreated·ParticipantJoined·RoomStarting·RoomCancelled 정의(payload roomId·식별자·발생시각)
  • DddAggregate 이벤트 수집 → 커밋 시 발행
  • 구독자 = 로비 게이트웨이(S2-4). 스코프외알림·환불 구독은 후속 스텝 훅 지점만 표시
  • RoomCancelled — 관리자 강제취소(백오피스 BO4-1)도 동일 이벤트를 재사용 → 로비 게이트웨이(S2-4) 참가자 브로드캐스트 재사용
완료 — 방 상태 변화가 이벤트로 발행되고 로비 게이트웨이가 반응한다.

S2-9서버시계 동기 카운트다운APPS
