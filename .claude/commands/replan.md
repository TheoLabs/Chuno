# 기획 변경 → 임팩트 전파

변경 내용: **$ARGUMENTS**

네 임무는 이 변경이 **의존성·연관 그래프 전체에 미치는 영향을 빠짐없이 분석**하고, **사용자 승인 후에만** 이슈·설계·도메인에 반영하는 것이다. 파일은 직접 쓰지 말고 서브에이전트에 위임한다.

## 1) 임팩트 분석 (issue-planner, 변경 모드)
`issue-planner` 서브에이전트에 `$ARGUMENTS`를 넘겨 **변경 임팩트 분석**을 시킨다(기존 docs: issue/mvp/domain/wireframes/README + `apps/core-api/CLAUDE.md`를 읽고 deps·크로스링크 그래프를 훑어). 반환받을 **체인지셋**:
- **직접 영향** + **하위 전파(전이적 deps)** + **크로스링크 전파**(도메인 엔티티/화면 참조처).
- **회귀**: 이미 완료/구현된 태스크가 재작업·재검증돼야 하는지.
- 항목별 **변경유형**(신규/수정/폐기/재검증/스코프변경) + 이유 + 바뀌는 완료기준/연관.
- **완결성 점검**(훑은 그래프·영향 없음 확인) + 가정/열린 질문.

## 2) 체인지셋 제시 + 승인 게이트
- 영향 표를 사용자에게 제시(대상·변경유형·이유·회귀 여부). **특히 이미 완료된 태스크의 재작업/재검증 필요를 눈에 띄게.** 승인 전에는 아무것도 건드리지 않는다. 조정은 반영해 다시 제시.

## 3) 반영 (승인 후에만)
- **docs-editor**: 영향받는 이슈(`issue/step*`·`mvp/step*`·`issue/index` 보드)·도메인(`domain/index` 카드/매핑/ERD)·크로스링크·이슈 카운트를 갱신. **재검증 필요 태스크는 완료 태그를 되돌려 부분/남음으로**, 폐기 태스크는 제거/스코프외 표기.
- **wireframe-designer**: 디자인이 바뀌는 화면을 반영하고 이슈↔와이어프레임 크로스링크를 맞춘다.
- 같은 파일 동시 편집 충돌 방지(겹치면 순차).

## 4) 요약 + 후속
적용된 체인지셋을 요약하고, **재검증(`/verify-task`)·재구현(`/chuno-mobile` 등)이 필요한 태스크 목록**을 명시해 다음 액션으로 잇는다.
