# 태스크 검증·리뷰 → docs 최신화

대상 태스크: **$ARGUMENTS**

네 임무는 위 태스크가 **실제로 구현/완료되었는지 검증·리뷰**하고, **검증으로 확인된 실제 상태를 docs에 반영**하는 것이다. 전부 통과가 아니어도 **부분적으로 완료된 항목은 완료로, 남은 항목은 남음/이월로** 정확히 표기한다 — 즉 docs를 항상 "검증된 현재 상태"에 맞춘다.
docs 수정은 반드시 `docs-editor` 서브에이전트에 위임한다 — **docs 파일을 직접 편집하지 마라.**

## 1) 태스크 스펙 파악
- `docs/issue/step*.html`에서 `$ARGUMENTS` 이슈 블록(예 `#S1-1`)을 찾아 **체크리스트 항목과 "완료 —" 기준**을 읽는다.
- 대응하는 `docs/mvp/step*.html`의 해당 항목도 확인한다.
- 태스크 성격에 따라 대상 코드/문서를 정한다: BACKEND→`apps/core-api`, APP→`apps/chuno-mobile`, DESIGN/OPS→관련 문서·설정.

## 2) 코드 리뷰·검증 (code-reviewer 에이전트에 위임)
- **코드 검증은 `code-reviewer` 서브에이전트(subagent_type: `code-reviewer`)에 위임한다.** 직접 파일을 일일이 대조하지 말고, 1)에서 파악한 **태스크 ID·완료 기준·대상 코드 범위·관련 설계 결정**을 프롬프트로 넘겨 리뷰를 맡긴다.
- code-reviewer는 읽기 전용으로 완료 기준을 실제 코드와 대조하고, 필요한 검증 명령(빌드/린트/테스트)을 돌려 근거를 확보한 뒤 **구조화된 리포트(판정 + ✅충족/⚠️갭/❗문제)** 를 반환한다.
- 외부 자원/키가 필요해 검증 불가한 부분은 그 사실을 명시하도록 한다.

## 3) 리뷰 리포트
- code-reviewer가 반환한 **판정과 ✅충족/⚠️갭/❗문제**를 사용자에게 그대로 간결히 전달한다(각 항목 파일 경로·근거 포함).

## 4) 판정 정리 (항상 docs에 반영)
게이트로 docs 갱신을 통째로 막지 않는다. 대신 각 항목을 **검증 결과에 따라 상태 분류**하고, 그 상태를 5)에서 docs에 반영한다:
- **완료** — 코드로 실증되고 완료 기준을 충족한 항목.
- **남음** — 미구현·부분구현, 또는 code-reviewer가 올린 미해결 문제(정확성 버그/설계 이탈).
- **이월** — 설계 결정으로 다른 태스크/스텝으로 미룬 항목(이월 대상 명시).
- **스코프 외** — 사용자 결정으로 이 태스크에서 제외한 항목.

그리고 **전체 판정**을 낸다:
- 모든 완료 기준 충족 + 미해결 문제 없음 → **태스크 통과(완료)** 로 표기.
- 하나라도 남음이 있으면 → **부분 완료**. 이때도 5)에서 **완료된 항목은 완료로, 남은 항목은 남음/이월로** docs를 갱신하고, 남은 작업·권장 조치를 사용자에게 보고한다(태스크 전체를 통과로 표기하지 말 것).

**절대 규칙**: 검증으로 충족을 확인하지 못한 항목을 완료로 표기하지 마라.

## 5) docs 최신화 (검증된 상태 반영 — 부분 완료 포함)
- `docs-editor` 서브에이전트(subagent_type: `docs-editor`)에 위임하여, 4)의 상태 분류를 반영해 최신화한다:
  - `docs/issue/step*.html`의 해당 이슈 — 항목별 **완료/남음/이월/스코프외** 상태 태그를 정확히 반영하고, 완료 기준(`.iaccept`) 문구도 현재 상태에 맞춘다.
  - `docs/mvp/step*.html`의 해당 항목.
  - `docs/issue/index.html`(이슈 보드) — 해당 이슈 행의 **완료 태그 유무를 전체 판정에 맞춰 판단·수정**한다(통과→완료 태그 부여, 부분/미완→완료 태그 제거 또는 부분 표기). 보드가 실제 상태와 어긋나지 않게.
  - 스키마/도메인이 바뀌었다면 `docs/domain/index.html`(엔티티 카드·매핑 표·ERD)도.
- docs-editor에는 **각 항목의 상태와 근거(확인된 실제 구현 사실)**, 이월 대상, 스코프 제외 사유를 구체적으로 전달하고, 기존 **다크+코랄 스타일·자체완결·상호링크** 규칙 유지를 지시한다.
- 완료 후 무엇을 어떤 상태로 갱신했는지 요약한다.
